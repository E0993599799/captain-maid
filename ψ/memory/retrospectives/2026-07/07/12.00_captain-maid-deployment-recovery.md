// ✅ CORRECT - Server layout wraps client components with context
export default async function LocaleLayout({ children, params }) {
  const { locale } = await params;
  const messages = (await import(`../../locales/${locale}.json`)).default;
  
  return (
    <html lang={locale}>
      <body>

      </body>
    </html>
  );
}

## Architecture Decisions

### 1. Removed force-dynamic in Favor of Static Generation
**Why**: force-dynamic + generateStaticParams() create incompatible rendering modes on Vercel.
**Alternative considered**: Make everything dynamic (rejected — kills performance)
**Decision**: Use generateStaticParams() for proper static generation

### 2. NextIntlClientProvider Over Server-Side Context
**Why**: Client components need i18n context during prerender. Server-side setup alone is insufficient.
**Alternative considered**: unstable_setRequestLocale in layout (rejected — breaks with static gen)
**Decision**: Wrap children with NextIntlClientProvider in layout, passing locale + messages as props

### 3. Translation Key Audit Before Code Fixes
**Why**: next-intl throws synchronously on missing keys; JavaScript fallbacks never execute.
**Alternative considered**: Add fallbacks where used (rejected — futile, throws first)
**Decision**: Audit ALL t() calls, verify keys exist in both locales BEFORE deploying

### 4. Serial Commit Strategy
**Why**: With cascading failures, one fix per commit allows easy identification of which change resolved the issue.
**Decision**: Commit 1 fix → build → test → next fix. Each commit is independently reversible.

## AI Diary (Vulnerable Reflection)

This session was humbling. I started confident that I understood Next.js deployment and next-intl. By 08:00, I'd been wrong four times in a row. The pattern of my mistakes:

1. **Overconfidence in fallbacks**: I kept assuming `t('key') || 'fallback'` would protect against missing keys. I didn't understand that next-intl throws *before* the JavaScript `||` operator can execute. This burned 90 minutes.

2. **Force-dynamic as a universal fix**: When things failed, I reflexively reached for `force-dynamic` thinking "make it dynamic, skip the hard static generation." This masked the real issue (missing context provider) and created new conflicts.

3. **Not auditing translations systematically**: Instead of doing a complete audit once, I fixed errors one by one as they appeared in build logs. If I'd audited all 49 keys upfront (took 10 minutes), I'd have saved 2 hours.

4. **[→ AGENT DECISION]** Trying to use `unstable_setRequestLocale` as the primary solution: I kept attempting to make this work when it was fundamentally incompatible with static generation. I should have recognized that a function labeled `unstable_*` was a red flag and sought the documented NextIntlClientProvider pattern from the start.

What I did right: Once I started the serial commit strategy (one fix per build), the feedback loop became clear. Each build showed exactly what broke next, making diagnosis mechanical rather than speculative.

## Honest Feedback (What Dragged)

**Friction Point 1: Understanding Next.js 15 + next-intl Interactions**
The combination of these two libraries has subtle, undocumented interactions. The docs for next-intl don't clearly state that client components *must* be wrapped with NextIntlClientProvider in a server layout. I inferred this after 3+ hours of trial and error. Recommendation: next-intl should have a Vercel-specific deployment guide showing the exact layout pattern.

**Friction Point 2: Vercel Build Error Messages Are Vague**
"Error occurred prerendering page" gives no details about *why* the page failed. The actual error (missing context for useTranslations()) required deploying an error boundary and reading browser console logs. Better error messages from Vercel would save 1-2 hours per session.

**Friction Point 3: Translation Key Discovery Process**
Without systematic grep + json.load validation, I can't know if a key exists until the build fails and reveals it. I wrote a Python audit script on the 8th attempt; running it first would have prevented 4 errors. Tooling for i18n key validation should be built into the test suite.

## Lessons Learned (Generalizable)

### Lesson 1: NextIntlClientProvider is Non-Negotiable
For ANY Next.js app using next-intl with client components, the server layout MUST wrap children with NextIntlClientProvider passing locale and messages. This is not optional or a nice-to-have — it's the boundary between working and broken. Test this pattern early.

**Application**: Add this to deployment checklist for any next-intl + Vercel project.

### Lesson 2: Static Generation Beats force-dynamic for Stability
force-dynamic + generateStaticParams() are incompatible on Vercel. Static generation with ISR is more stable, faster, and less likely to fail. Use force-dynamic only for routes that genuinely can't be pre-rendered (e.g., dynamic API endpoints).

**Application**: Default to static generation. Only add dynamic rendering when proven necessary.

### Lesson 3: Audit Translations Before Deploying
next-intl throws synchronously on missing keys. JavaScript fallbacks don't protect you. Before deploying:
- grep all `t('...')` calls
- Verify every key exists in both locale files
- Use validation tooling (json.load + dot notation check)

**Application**: Add i18n audit to pre-deploy checklist. Make it automated.

### Lesson 4: Serial Commit Strategy for Cascading Failures
When multiple systems interact (rendering, i18n, routing), failures cascade. One atomic fix per commit + build test allows clear feedback on what resolved what. This is faster than trying to guess multi-system fixes.

**Application**: For complex systems, commit frequently and test after each change. The feedback loop is the debugging tool.

### Lesson 5: Deprecated APIs Are Red Flags
Functions labeled `unstable_*` or `deprecated` are not fallbacks or alternatives. They're either going away or have known issues. When one doesn't work, don't persist — look for the documented, stable pattern instead.

**Application**: Treat deprecated APIs as "do not use" rather than "use as last resort."
