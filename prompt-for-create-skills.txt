SECTION 4 — CONTENT REQUIREMENTS PER FILE
=========================================

A) /skills/performance.md must include:

* Mission: make pages fast, responsive, stable, and production-ready
* Scope: every page/component change, especially landing pages, dashboards, forms, media-heavy pages
* Core Web Vitals targets:

  * LCP should be good
  * INP should be good
  * CLS should be good
  * If exact measurement is unavailable, clearly say not measured
* Image rules:

  * Use next/image where appropriate
  * Set width/height or stable aspect-ratio
  * Avoid oversized source images
  * Compress images before use when needed
  * Avoid loading hidden desktop/mobile duplicate images
  * Lazy-load non-critical images
  * Prioritize only the true LCP image
* Font rules:

  * Use next/font or local fonts
  * Limit font families and weights
  * Avoid external blocking font chains
  * Avoid layout shift from late font loading
* JavaScript rules:

  * Avoid unnecessary "use client"
  * Keep components server-rendered by default
  * Use client components only for actual interaction/state/browser APIs
  * Avoid importing heavy libraries for simple UI
  * Use dynamic import for heavy client-only components
  * Avoid large icon imports
* Animation rules:

  * Keep animation subtle and meaningful
  * Avoid layout-triggering animation
  * Prefer transform and opacity
  * Respect reduced-motion
  * Avoid infinite animation unless decorative and very lightweight
* Bundle rules:

  * Check package imports
  * Avoid duplicate packages
  * Avoid client-bundling server-only utilities
  * Use bundle analyzer when available
* Mobile rules:

  * Test mobile first
  * Avoid huge hero height on mobile
  * Ensure tap targets are usable
  * Avoid horizontal overflow
* Performance proof:

  * commands run
  * build result
  * Lighthouse/Core Web Vitals if measured
  * before/after notes if available
  * unmeasured risks clearly stated

B) /skills/security.md must include:

* Mission: protect app, users, data, secrets, and production environment
* Scope: every auth route, API route, server action, form, upload, dashboard, admin page, and deploy
* OWASP-based threat model
* Environment rules:

  * Never expose secrets in frontend
  * NEXT_PUBLIC_* means browser-visible
  * Never put Supabase service role key in frontend
  * Do not paste secrets into chat/logs
  * Separate development, preview, and production envs
* Auth rules:

  * Do not rely only on hidden UI
  * Enforce auth on server
  * Check role/permission on server
  * Protect admin routes
  * Protect API routes
* Input validation:

  * Validate all server inputs
  * Validate API body/query/params
  * Validate server action input
  * Use schema validation when possible
* Output sanitization:

  * Avoid dangerouslySetInnerHTML
  * Sanitize user-generated HTML if unavoidable
  * Prevent XSS
* API rules:

  * Check method
  * Check auth
  * Check authorization
  * Validate input
  * Rate-limit or abuse-protect sensitive routes when possible
  * Return safe errors
* File upload rules:

  * Limit file type
  * Limit file size
  * Avoid trusting filename
  * Avoid path traversal
  * Store outside executable paths
* Database rules:

  * Use parameterized queries or safe ORM/client
  * Use RLS where applicable
  * Keep service role key server-only
  * Never trust client-side filters for authorization
* Header rules:

  * Content-Security-Policy
  * X-Content-Type-Options
  * Referrer-Policy
  * Permissions-Policy
  * frame-ancestors or X-Frame-Options
  * Strict-Transport-Security when appropriate
* Logging rules:

  * Log enough for debugging
  * Never log tokens/passwords/secrets
  * Avoid exposing stack traces to users
* Security proof:

  * routes checked
  * env exposure checked
  * auth checked
  * API/server actions checked
  * headers checked
  * remaining risks

C) /skills/vercel-deploy.md must include:

* Mission: safely deploy, verify, debug, rollback, and document Vercel deployments
* Scope: any preview deploy, production deploy, env change, domain change, rollback, or deploy debug task
* Package manager detection:

  * pnpm-lock.yaml means pnpm
